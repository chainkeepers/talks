consensus

each validator pre-votes and gossips the pre-vote

more than 2/3 pre-votes → vote and gossips

more than 2/3 votes → precommit and gossip

more than 2/3 votes → commit and gossip

upon timeout, assume no vote

non (vote ⇒ pre-vote) → you're slashed

sei

limitations

each timeout slows processing

wait for proposal, pre-vote and vote

gossip is resilient, but not efficient

leader can fall behind

invalid blocks are proposed

transactions don't get included

sei

solution

about

based on Diem / Libra (Wikipedia)

DAG mempool: Narwhal+Bullshark (arXiv, arXiv)

has traits of Solana, Avalanche, Bitcoint

intuition is that the model scales massively

sui

move

transaction is a Move call

[input objects, …] → [output objects, …]

shared objects: anyone can interact

owned objects: only sole owner can interact

only a single concurrent transaction for owned objects

sui

consensus

anyone can produce blocks: a DAG blockpool

create a total ordering of partially ordered blocks

[input(v), …] → outputs(v+1)

consensus may order only part of the DAG

owned objects skip consensus altogether

steady-state ⇄ fallback

sui

submission

client → tx → 2/3+ validators → certificate

certificate → 1/3+ validators → state

sei

limitations

not implemented fully yet

only owned objects skip consensus

all shared objects undergo total-order consensus

execution is paralelized fully → cheap gas
